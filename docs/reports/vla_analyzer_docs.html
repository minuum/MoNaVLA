VLA-Analyzer — 개요

로컬 HTTP 서버 (localhost:8888) 및 V5 데이터셋 구조 설명

VLA-Analyzer란?

VLA-Analyzer는 MoNaVLA V5 데이터셋의 grounding 결과를 시각적으로 탐색하기 위한 로컬 정적 웹 도구다. Python의 http.server를 사용해 localhost:8888에서 서빙되며, 별도 서버 설치 없이 즉시 실행된다.

서버 실행 방법

# v5_data_bak 디렉토리에서 실행
cd ROS_action/v5_data_bak
python -m http.server 8888

# 브라우저에서 접속
http://localhost:8888/v5_viewer.html          # 에피소드 뷰어
http://localhost:8888/vla_analyzer_docs.html  # 이 문서

V5 프로젝트 맥락

V5 데이터셋은 MoNaVLA의 BBox-Centric 검증 방법론을 위해 수집된 데이터다. 로봇이 바구니를 향해 접근할수록 bbox가 커지고 화면 중앙으로 수렴해야 한다는 가설을 검증한다. 이 문서의 4~5페이지에서 bbox 해석 기준과 검증 방법론을 상세히 설명한다.

image_to_text_projection 오염 문제

Kosmos-2의 멀티모달 아키텍처는 세 구성 요소로 이루어진다. 핵심은 비전 feature를 언어 모델 토큰 공간으로 매핑하는 브릿지 레이어다.

Image Encoder (Vision) ← frozen 유지
↓
image_to_text_projection ← action 학습으로 덮어씌워짐
↓
Text Decoder (Language Model)
↓
Grounding / Text Output

오염 메커니즘

Google-robot pretrained과 V4 LoRA는 action prediction을 위해 fine-tune된 모델이다. 학습 목표는 이미지 + 지시문으로부터 연속 action vector ([linear_x, linear_y, angular_z, ...])를 예측하는 것이다.

이 과정에서 image_to_text_projection이 action feature 공간에 맞게 재학습되면서 원래의 텍스트 생성 경로가 파괴된다.

학습 전학습 후
image features → text tokens (grounding / caption 생성) image features → action features (텍스트 생성 불가)

LoRA 우회 시도가 실패하는 이유

backbone.base_model.model.generate()를 통해 LoRA 레이어를 건너뛰어도 동일한 garbage 출력이 발생한다.

  • LoRA adapter 자체도 action prediction 방향으로 수렴됨
  • image_to_text_projection은 LoRA가 아닌 full fine-tune으로 덮어씌워짐
  • 기반 모델 가중치 자체가 이미 오염된 상태
결론: fine-tuned 체크포인트에서 원본 grounding 능력을 복구하는 것은 현실적으로 불가능하다. Pure HF 모델을 별도로 로딩하는 것이 유일한 해결책이다.

Grounding 조사: 프롬프트 비교

V5 로봇 이미지 (복도의 회색 바구니) 기준, 동일 이미지 비교 테스트

전체 프롬프트 비교표

프롬프트 검출 entity 이름 BBox 유효 비율 평가
<grounding>An image of a robot. Where is the gray basket? Answer: "The room" (0,0) → (1,1)
전체화면		 1.8% 실패 — 전체화면 hallucination
<grounding><phrase>gray basket</phrase> <patch_index_493>...
entity 이름 깨짐		 (0.42,0.48) → (0.58,0.70) 가변 bbox는 정확하나 entity 이름 손실
<grounding>An image of <phrase>gray basket</phrase> <patch_index_493>...
entity 이름 깨짐		 (0.42,0.48) → (0.58,0.70) 가변 위와 동일
<grounding>A gray basket is located at "a white wall" 배경 위치 낮음 실패 — 배경 오검출
<grounding>The gray basket is at "the gray box" / "The basket" (0.33,0.45) → (0.55,0.83) 100% 최선 선택

최종 채택 프롬프트

prompt = "<grounding>The gray basket is at"

채택 이유 (4가지)

1. Entity 이름 정확성

"the gray box" 또는 "The basket" — 바구니를 올바르게 지칭한다. 분류 필터에서 키워드 매칭이 가능하다.

2. BBox 정확성

전체화면 bbox가 아닌 실제 객체 위치를 가리키는 좌표를 반환한다.

3. 유효 검출률 100%

테스트된 모든 프레임에서 non-fullscreen bbox를 반환한다. 다른 프롬프트는 전체화면 또는 배경 오검출이 빈번했다.

4. Completion-style prompting

문장을 완성하도록 유도하는 방식을 사용한다. 모델이 specific location을 생성하려는 경향을 활용해 더 정확한 grounding이 유도된다.

왜 <phrase> 태그 방식은 실패했는가

<phrase>gray basket</phrase> 방식은 Kosmos-2 내부에서 entity 이름을 <patch_index_N> 형태의 포지셔널 토큰으로 치환한다. 이는 grounding token 생성 방식의 특성으로, entity 이름 자체가 손실된다. 색상 분류/필터링 단계에서 entity 이름을 키워드로 사용하는 파이프라인과 호환되지 않는다.

최종 파이프라인 요약

# 모델 로딩
model = AutoModelForVision2Seq.from_pretrained(
    "microsoft/kosmos-2-patch14-224"
)

# 프롬프트 설정
prompt = "<grounding>The gray basket is at"

# 후처리 필터
if bbox_area > 0.90:
    skip()  # fullscreen hallucination 제거

# entity 이름으로 색상 분류 (4페이지 참조)
color = classify_entity(entity_name)

BBox 해석 가이드

색상 분류 기준, fullscreen 필터, BBox convergence 패턴

색상 분류 기준

Grounding 결과의 entity 이름을 키워드 매칭으로 분류한다. 뷰어에서 bbox 테두리 색상으로 표시된다.

초록 (Green) — 바구니 확실

entity 이름에 "basket", "box", "container" 포함.
바구니를 정확히 검출한 것으로 판단.

노랑 (Yellow) — 가능성 있음

entity 이름에 "gray", "grey" 포함.
회색 물체, 바구니일 가능성 있으나 확정 불가.

주황 (Orange) — 배경 오검출

위 조건 미해당. "wall", "floor", "air conditioner" 등.
배경을 잘못 검출한 것으로 판단.

Fullscreen 필터: bbox 면적이 전체 이미지의 90% 이상 (area > 0.90)인 경우 hallucination으로 판단하여 표시하지 않는다. 이는 "The room" 같은 전체화면 entity를 제거하기 위한 필터다.

BBox 면적 계산

# BBox 형식: (x1, y1, x2, y2), 정규화된 0~1 좌표
area = (x2 - x1) * (y2 - y1)

# 예시
bbox = (0.33, 0.45, 0.55, 0.83)
area = (0.55 - 0.33) * (0.83 - 0.45)  # = 0.22 * 0.38 = 0.084

# fullscreen 판정
if area > 0.90: skip()

BBox Convergence 패턴 — ep3 예시

ep3 (Fixed Center Path)에서 로봇이 바구니를 향해 접근할수록 bbox가 어떻게 변화하는지를 보여주는 실제 데이터다. 이 패턴이 V5 BBox-Centric 검증의 핵심 가설이다.

f00-f04 주황 "white wall" 계열 entity 배경 좌표 — 바구니 미검출 —
f05 초록 "The box" (0.33, 0.45) → (0.55, 0.80) 0.077
f06 초록 "The basket" (0.33, 0.45) → (0.55, 0.83) 0.083
f09 초록 "The basket" (0.20, 0.42) → (0.52, 0.95) 0.170
f10 초록 "The basket" (0.14, 0.42) → (0.52, 0.98) 0.211
f11 초록 "The basket" (0.11, 0.42) → (0.55, 0.98) 0.246

패턴 해석

  • 초기 프레임 (f00-f04): 바구니가 멀어 배경(흰 벽)이 오검출됨
  • 중간 프레임 (f05-f06): 바구니가 시야에 들어오기 시작, "The box" / "The basket" 검출
  • 후기 프레임 (f09-f11): 로봇이 접근할수록 bbox가 더 크고 하단으로 확장
  • 수렴 방향: x 좌표 좌우로 확장 + y2가 1.0(하단 끝)에 근접 → 바구니가 화면 하단을 채움
핵심 관찰: bbox 면적이 에피소드 진행에 따라 단조 증가하고, bbox 중심이 이미지 중앙 하단으로 수렴하는 것이 "올바른 접근 행동"의 visual signature다. 이것이 V5 BBox-Centric 검증의 기반이 되는 패턴이다.

V5 BBox-Centric 검증 방법론

무엇을 검증하는가, 성공 기준, 뷰어 사용법

검증의 목적

V5 검증은 MoNaVLA 모델이 바구니를 향해 올바르게 접근하는 행동을 학습했는지를 시각적으로 확인하기 위한 방법론이다.

기존 검증 방식(action vector 수치 비교)은 행동의 의미를 직관적으로 파악하기 어렵다. BBox-Centric 방법론은 "로봇 카메라에서 바구니 bbox가 어떻게 변화하는가"를 ground truth로 사용한다.

검증 대상

검증 항목측정 방법
BBox 크기 증가 에피소드 진행에 따라 bbox 면적이 단조 증가하는지 확인
중앙 수렴 bbox 중심 x좌표가 0.5 (화면 중앙)에 수렴하는지 확인
하단 채움 에피소드 후반부에서 bbox가 화면 하단 절반을 채우는지 확인
색상 전환 주황(오검출) → 초록(바구니 검출)으로 에피소드 내에 전환 발생 확인

성공 기준 (Success Criteria)

  • 에피소드 후반부 50% 이상의 프레임에서 초록 bbox 검출
  • 마지막 프레임의 bbox 면적 > 0.15 (바구니가 화면의 15% 이상 차지)
  • bbox 중심 x좌표: 에피소드 종료 시 |center_x - 0.5| < 0.15 (중앙 ±15% 이내)
  • bbox y2 좌표: 에피소드 종료 시 y2 > 0.80 (화면 하단 80% 이하까지 닿음)

뷰어 사용 방법

v5_viewer.html에서 에피소드별 grounding 결과를 확인할 수 있다.

뷰어 주요 기능

  • 에피소드 선택: 좌측 패널에서 ep0 ~ ep49 중 선택
  • 프레임 탐색: 프레임별로 넘기며 bbox 변화 확인
  • 색상 범례: 초록/노랑/주황으로 검출 품질 즉시 파악
  • BBox overlay: 실제 이미지 위에 bbox가 오버레이로 표시됨
  • entity 이름 표시: bbox 위에 Kosmos-2가 생성한 entity 이름 표시

V5 이후 개선 계획

단기 (V5 검증 완료 후)

  • BBox convergence 패턴을 자동 감지하는 알고리즘 추가
  • 에피소드별 convergence score 계산 및 랭킹
  • 오검출 프레임 패턴 분석 → 데이터 수집 가이드 개선

중기 (모델 학습 개선)

  • image_to_text_projection을 frozen으로 유지하는 학습 설계 검토
  • Grounding 신호를 action 학습의 auxiliary loss로 활용
  • BBox-guided attention: 바구니 위치 정보를 action prediction에 직접 주입
